### The feature hunt: user-suggested features, thoroughly hunted and lens-explained
- **Description**: user request — the human should be able to suggest a feature the model didn't find, and the connected model then hunts the codebase for it, flagging it if already stated. `propose_feature` (`cms/feature_discovery.py`) upgraded from a single-shot mapping into a four-stage hunt: (1) **Already stated?** — deterministic overlap detection (`existing_overlap`: name/alias token match + member overlap between existing features and the ranked evidence) that fires even under mock and always surfaces alongside any model judgment, provenance-tagged `graph` vs `llm`; (2) **thorough search** — intent-ranked hits (top 14) expanded one hop through real CALLS/IMPORTS/PART_OF edges (`_neighbourhood_edges`), plus the full existing-feature catalog in the prompt so the model reasons over connections, not keywords; (3) **grounded mapping** — one LLM pass returns verdict (`already_covered | partial_overlap | new | not_found`), member roles (entry/core/support), connections, and an ordered mechanism explanation; every claim validated (unknown member ids and feature names dropped, bad roles degraded), and `graph_connections` merges edge-grounded connections so the connection list never rests on model assertion alone — graph evidence overrides the model's `via`; (4) **human confirmation** unchanged. UI: the hunt panel shows an amber "Already stated" banner with jump links to the existing feature, role-chipped members, provenance-chipped connections, and a "How it works" block tagged `data-lens` — so the explanation automatically reads at whatever comprehension level the user has selected on the Lens slider (schoolchild through specialist), composing with Human View resolution. Same payload over MCP `discover_feature` (agents told to check `existing` before proposing).
- **Verification**: `test_feature_discovery.py` grew to 8 tests — deterministic already-stated flag under mock, `already_covered` verdict suppresses duplicate candidates, hallucinated existing/connection features dropped, bad roles degraded, edge-grounded `graph_connections` asserted against a real cross-feature CALLS edge, hunt prompt carries catalog + neighbourhood. ### Embedded Memory Anchors (Custom Auto-Finding Structuring)
- **Description**: Allows developers to explicitly guide the memory layer with special markers in source code. These are extracted during scanning/summarization and enrich graph nodes with high-level intent, responsibilities, connections, and architecture notes that pure AST analysis cannot infer.
- **Proposed Syntax**:
  ```python
  # @memory:feature:UserAuthentication
  # @memory:connects:LoginFlow, TokenService, RateLimiter
  # @memory:summary:Handles JWT issuance, refresh, and blacklisting with device fingerprinting.
  def login_user(...):
      ...

  # Or block-level:
  # === @memory:module:GraphLayer ===
  # Purpose: Maintains the runtime knowledge graph
  # Key flows: scan → parse → summarize → persist
  class MemoryEngine:
      ...
  ```
- **Integration**:
  - Parser extracts `@memory:` tags and boosts graph nodes.
  - Enriches lightweight summaries with developer-curated understanding.
  - Improves AI navigation and self-bootstrapping by combining auto-analysis with human intent.
- **Status**: ✅ IMPLEMENTED (2026-07-04) in `cms/anchors.py`.
  - Extraction uses Python's `tokenize` module (real COMMENT tokens only), so anchor-like text inside strings/docstrings is ignored.
  - Line-form groups attach to the def/class starting within 6 lines below; `module` tags and orphan groups attach to the file node.
  - `connects` values are comma-split lists; block-form (`===`) plain comment lines become `notes`.
  - Anchors are stored on graph nodes (`anchors` attr), fed into LLM prompts as authoritative intent, included in mock summaries, shown in `cms query` output, and boosted in ranking (+2.5/token — above generated summaries at +2.0).
  - CMS's own source is dogfooded with anchors: CleanDirectoryScanner, KnowledgeGraphConstruction, SummaryGenerator, QueryEngine, MemoryAnchors.

### Self-Bootstrapping Core
- The system will first analyze its own directory (`./` or `./cms/`) to generate `.memory/` layer.
- Uses its own clean tree, graph, and summaries to inform future improvements.
- Creates a compounding intelligence flywheel with top-tier models.

### Core Vision Reminders
- Clean scanner with strong ignore patterns (no node_modules, venv, build artifacts, etc.).
- Knowledge graph with embedded lightweight summaries (low-res pseudo-code + line numbers + connections).
- Queryable memory layer for AI agents to understand full codebase context efficiently.
- Goal: State-of-the-art codebase understanding tool that improves itself over time.
